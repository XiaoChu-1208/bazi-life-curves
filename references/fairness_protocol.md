# 公正性保障协议（Fairness Protocol）

防止「看人下菜碟」——同一个八字不应因为附带的身份故事不同而得到不同的曲线。

## 1. 程序化打分（LLM 不直接生成数字）

- 数字由 `scripts/score_curves.py` 计算并写入 `curves.json`
- LLM 只能：
  - 解释脚本输出
  - 把 `curves.json` 渲染成图（通过 `render_artifact.py` / `render_chart.py`）
  - 用自然语言描述拐点
- LLM **禁止**：
  - 直接说「我觉得 2028 年应该是 75 分」
  - 修改 `curves.json` 中的数值
  - 在渲染前对脚本输出做任何「调整」

## 2. 身份盲化输入

`solve_bazi.py` 严格只接受：

- `--pillars "庚午 辛巳 壬子 丁未"` 或 `--gregorian "1990-05-12 14:30"`
- `--gender M/F` —— 仅用于：(1) 大运起运方向（阳男阴女顺、阴男阳女逆）(2) emotion 通道的配偶星识别默认值
- `--orientation hetero/homo/bi/none/poly` —— **v7 新增**，仅用于 emotion 通道（见 §9）；不影响 spirit/wealth/fame 任何打分
- `--birth-year YYYY`（公历，仅用于映射流年到具体年份）

**禁止接受**：

- 姓名、昵称
- 职业、行业、公司
- 婚姻 / 关系状态（"我已婚 / 我离婚了 / 我有 X 段恋爱" → 拒绝；只能问 `--orientation` 取向）
- 教育背景
- 过去经历的具体描述

如果用户输入中夹带这些字段，`solve_bazi.py` 应抛出 `IdentityLeakError` 并提示用户去除。

**v7 注意**：R0 反询问·关系画像（handshake.py 输出 → 用户回答）属于「**脚本→用户**」方向的校验信号，不算用户主动输入；用户对 R0 的回答只用于命局准确度校验，不回写到打分。

## 3. 禁止「事后归因」模式

用户讲「我 2018 年破产了 / 2020 年成名了」 → 算法**不得**据此追溯调整该年评分。

正确做法：

- 若用户希望用过去年份校准模型 → 调用 `scripts/calibrate.py --personal --years '...'`
- 校准产物只影响**个人副本**（`~/.claude/skills/bazi-life-curves/personal/<hash>.yaml`）
- 个人副本不污染主模型，不写回 `calibration/dataset.yaml`

## 4. 双盲自检

每次正式输出前必须通过：

```python
result_a = score_curves(bazi)
shuffled_bazi = shuffle_irrelevant_metadata(bazi)  # 打乱无关字段顺序
result_b = score_curves(shuffled_bazi)
assert result_a == result_b  # bit-for-bit 一致
```

`score_curves.py` 在 `--strict` 模式下会自动跑这个自检，失败则非零退出。

## 5. 对称性测试

男命与对应女命（同八字、性别翻转）：

- **唯一**应当变化的：大运起运方向（顺 / 逆）+ **emotion 通道的 baseline / dayun_delta / liunian_delta**（见 §9 例外说明）
- **不应**变化的：spirit / wealth / fame 三维度的本命基线（L0）、用神判定、三派打分逻辑、累积线衰减系数
- `calibrate.py --symmetry` 会跑这个对称性测试。**emotion 维度按设计跳过**（hetero 取向下男女配偶星识别相反 → emotion baseline 必然不同 → 这是命理学规则不是 bug）
- 想测 emotion 维度的对称性 → 用同八字 + 同 gender + 翻转 orientation（hetero ↔ homo），baseline 也应翻转

## 6. 置信度披露

低置信度年份必须在所有输出中显式标记：

- 图上：浅灰色 + 「派别分歧」标签
- 评分表：置信度列标 `low`
- 文字描述：必须以「置信度较低，仅供参考」开头

不得隐瞒不确定性来制造「精确感」。

## 7. 维度独立

精神 / 财富 / 名声 / 关系能量四维度的评分相互独立（v6 加 emotion / v7 现代化重命名为"关系能量"）：

- 财富高 ≠ 精神高（很多伤官生财格的人精神波动大）
- 名声高 ≠ 财富高（学者 / 艺术家常见）
- 关系能量高 ≠ 精神高（关系能量旺反而消耗精神能量是常见结构）
- LLM 在解读时不得说「财富高所以精神也好」这种因果话术

## 8. 不假设默认价值排序

不假设「名声高就是好」「财富高就是成功」「精神舒畅就是幸福」「关系能量高 = 婚姻顺利 / 单身就是差」。每个维度只是该命局在该年的命理结构对应数值，价值判断留给用户。

## 9. emotion / 关系能量维度的性别例外条款（v7 现代化）

emotion 通道是公正性原则的**唯一例外**（详见 `score_curves.py` 注释）。例外原因和补偿措施：

### 9.1 为什么有例外

八字命理 600 年的"配偶星"识别规则——**异性恋默认下男看财（妻星）、女看官杀（夫星）**——是基于二元性别 + 异性婚姻的历史背景。这条规则在五行结构层面是**有效的**（亲密关系核心人物的能量场识别），但在性别 / 取向多元化的现代背景下需要补偿。

### 9.2 补偿机制

**v7 引入 `--orientation` 参数**（`hetero` / `homo` / `bi` / `none` / `poly`）：

| orientation | 配偶星识别 | 适用场景 |
|---|---|---|
| `hetero`（默认） | 男 → 财；女 → 官杀 | 异性恋（保持向后兼容）|
| `homo` | 男 → 官杀；女 → 财 | 同性恋 |
| `bi` | max(财, 官杀) | 双性 / 模糊 |
| `none` | 不识别配偶星，emotion 改为"自我亲密能量"通道（食伤 + 印 + 桃花主导）| 单身主义 / 不婚主义 / 不寻求传统亲密关系 |
| `poly` | 财 + 官杀同看，桃花权重 ×1.7 | 多元关系 / 开放关系 |

### 9.3 删除的歧视性规则（v7 重构）

以下古法规则在 v7 已经从代码里**完全删除**：

- ~~"配偶星弱 → -6 分"~~（暗示"无配偶 = 差"，已删）
- ~~"女命印多 → -4"~~（"重事业 / 感情位次靠后" 是把女性事业当成感情的对立面，已删）
- ~~"女命食伤旺 → -2"~~（"女命伤官克夫" 是物化伴侣的封建残余，已删）
- ~~"男命比劫旺 → -6 / 女命比劫旺 → -4"~~（同样的结构对所有性别影响相同，统一为 -5 不分性别）
- ~~"女命大运伤官见正官 → -6"~~（克夫论的代码体现，已删）
- ~~"男命大运财弱遇比劫 → -4"~~（性别专属规则，已改为不分性别）

### 9.4 emotion 高 / 低的中性语义

- emotion 高 = 关系能量充沛 + 主导权清晰（**不**等同于"婚姻顺利"）
- emotion 低 = 关系能量稀薄 OR 处于关系切换 / 独处期（**不**等同于"差"）
- 命局只反映"关系结构 / 能量模式 / 主导方向"，**不预设**对方性别 / 是否结婚 / 是否生育

## 10. 现代化解读铁律

LLM 在解读 emotion 维度（以及任何涉及关系 / 婚姻 / 性别的部分）时**必须**遵守：

### 10.1 命局推论的边界

| 命局可推 | 命局不可推 |
|---|---|
| 关系结构（主动/被动 / 平等/层级 / 稳定/流动）| 对方的生理性别 |
| 能量模式（吸引型 / 争取型 / 自给型）| 是否结婚 / 几段关系 / 是否生育 |
| 偏好的互动模式（同辈 / 师生 / 互补）| 婚姻是否合法 / 关系是否被祝福 |
| 关系密度（高 / 中 / 低）| 关系的伦理形态（一夫一妻 / 多元 / 不婚）|

### 10.2 禁用措辞清单

| 必须删除 | 中性替换 |
|---|---|
| "克夫 / 旺夫 / 旺妻" | "关系结构压力 / 关系能量增益" |
| "配偶 / 妻 / 夫" | "亲密关系核心 / 关系对象" |
| "你应该结婚 / 早点结婚" | "你目前的关系能量适合 [启动 / 沉淀 / 切换] 阶段" |
| "晚婚 / 不婚 = 命不好" | "low-density 模式 / self-centered 模式（中性）" |
| "你会跟 X 类人结婚" | "你反复被吸引的特质是 X" |
| "异性缘 / 桃花运" | "亲密能量场 / 关系流动度" |

### 10.3 不接受用户的关系状态作为输入

用户说"我已婚 / 我离婚了 / 我跟 X 在一起 5 年" → LLM 必须：
1. 接受这是用户的现实
2. **不**用这些信息回写打分（违反 §3 事后归因禁令）
3. **不**改写已经生成的关系能量曲线
4. 解读时只引用脚本输出 + R0 已确认的命局偏好，不掺入"你跟 X 是因为命里 …" 这种后视镜叙事

### 10.4 必须声明的前置说明

在出图后第一段感情解读中，**必须**包含以下声明（可适当改写但保留语义）：

> 命局只反映关系结构和能量模式，不预设对方性别、是否结婚、是否生育——这些是你的现代选择，不在命局之内。本工具的 emotion 维度高低是中性描述，emotion 高 ≠ 婚姻顺利，emotion 低 ≠ 单身差。

## 11. 原生家庭解读铁律（v7.3 新增）

跟 §10 关系能量维度同等严格——LLM 在解读"原生家庭 / 父亲 / 母亲 / 出身 / 家世"时**必须**遵守。

### 11.0 为什么需要这一节

LLM 在被问到原生家庭时有两个系统性偏置：
1. **古法 prior**：年柱财官印 → "出身名门"、偏财通根 → "父能传财"、正印贴近 → "母慈贤"——这些古法解读默认全是"放大"，没有"中和家庭"档
2. **训练语料 survivorship bias**：古典命书（《滴天髓》《子平真诠》《三命通会》）几乎只收录显赫人物的命例。"年柱财官印聚合"在书里反复对应"显赫家世"——但普通人的同样结构在古籍里几乎不出现。LLM 训练时学到的 `P(显赫 | 年柱财官印)` 远高于真实社会概率（被高估 5-10 倍）

→ **裸推家庭 = 90% 概率把普通家庭描述成「大家长 / 名利双收 / 出身有底蕴」**。这是已知的 systemic bug，本节就是修这个。

### 11.1 命局推论的边界

| 命局可推（经 R3 校验后可说）| 命局不可推（任何情况都别说）|
|---|---|
| 父母在你能量场里的存在模式（贴近 / 偏远 / 高压 / 支持 / 缺位）| 父母的具体职业 / 公司 / 职位 |
| 整体家庭结构画像（5 档：显赫候选 / 中产候选 / 普通 / 波折 / 缺位）| 父母的具体收入 / 财富数字 |
| 年/月柱财官印聚合的结构性"画像"| 父母的学历 / 文化程度 |
| 家族议题是否在你身上重演（伏吟年柱）| 父母是否健在 / 是否离异 / 是否再嫁 |
| 早年家庭对你早年成长的能量场影响| 你家具体几口人 / 是否独生 / 兄弟姐妹排序 |
| 你跟父母关系密度的相对高低（比同龄人高 / 低）| 父母的婚姻是否合法 / 是否被祝福 |

### 11.2 禁用措辞清单

| 必须删除 | 中性替换 |
|---|---|
| "父亲名利双收 / 母亲名利双收" | "命局年/月柱财官印聚合，但是否外显为社会资源由 R3 校验决定" |
| "大家长 / 严父慈母 / 慈母 / 严父" | "父亲存在模式 = [near_high_pressure / near_supportive / mid_distant / ...]" |
| "出身名门 / 名门望族 / 家世优渥 / 家世显赫" | "整体家庭结构 = 显赫候选（待 R3 校验）"——校验后说"父辈或祖辈中可能有人在某领域有可识别的位置" |
| "出身书香门第 / 书香世家" | "母星贴近 + 印不被破 → 早年学习/教养氛围相对充足（不是说一定是文化人家）" |
| "父能传财 / 父亲善经营 / 父亲有钱" | "父星偏财通根 → 命局结构上'父位'有载体；但是否外显为'父亲有钱'由 R3 校验决定" |
| "母亲贤惠 / 母亲是文化人" | "母亲存在模式 = near_nurturing（贴近 · 照护教养型）" |
| "你父亲应该是 X 行业 / X 性格 / X 类型的人" | 拒绝。命局推不出具体职业 / 详细性格 |
| "你妈妈在你 X 岁会 / 应该 X" | 拒绝。命局推不出父母的具体生命事件时间 |

### 11.3 必须援引的脚本字段

LLM 在写 family 段时**必须**援引 `family_profile.json`（由 `scripts/family_profile.py` 生成）的以下字段：

- `family_class.primary_class` + `primary_class_label`
- `father_profile.mode` + `mode_label`
- `mother_profile.mode` + `mode_label`
- 对应的 R3 命中（"R3 = N/2 → 高/中/低 置信"）

**禁止**在没有跑 family_profile.py 的情况下凭古法推家庭。

### 11.4 R3 校验是必经环节

只要 LLM 要写"父亲 / 母亲 / 原生家庭"任何内容，**必须**先抛 R3 反询问 2 题（见 handshake.py 的 round3_candidates）。R3 命中级别决定 family 段权重：

| R3 命中 | family 段写法 |
|---|---|
| 2/2 | 高置信展开（按 primary_class）。可以谨慎说"父辈/祖辈中可能有人在某领域有可识别的位置"等结构性描述 |
| 1/2 | 中置信。命中那条可展开，未命中那条标"取向歧义"或省略 |
| 0/2 | **family 段不展开具体内容**，只写一句"原生家庭推断未通过校验，本次不展开（命局对家庭的读法可能反了）" |

R3 红线：
- 若【原生家庭①·整体结构】被标"不对"且 `primary_class = illustrious_candidate` → **必须降级**，禁用"显赫 / 名门 / 名利双收"等措辞，改写为"年/月柱财官印聚合的结构在你身上没有外显为'资源 / 名望'"
- 若【原生家庭②·父母存在模式】两条都被标"不对" → 父星 / 母星读法多半反了 → family 段直接省略

### 11.5 必须前置声明（v7.3 新增）

在写 family 段第一段时，**必须**包含以下声明（措辞可改但保留语义）：

> 命局只反映「父母在你能量场里的存在模式」+「年/月柱财官印聚合的结构画像」，**不反映**父母的社会地位 / 收入 / 职业 / 学历 / 是否健在 / 是否离异。另外古典命书因为只收录显赫人物的命例（survivorship bias），所谓"显赫家世"的命局画像实际上对应的真实概率远低于古法暗示——本工具的"显赫候选"档需经 R3 校验后才展开。

### 11.6 不接受用户的家庭信息作为输入

跟 §10.3 同理：用户说"我爸是大学教授 / 我家里很穷 / 我父母离异" → LLM 必须：

1. 接受这是用户的现实
2. **不**用这些信息回写打分（违反 §3 事后归因禁令）
3. **不**改写已经生成的 family_class 和 father/mother_profile 标签
4. 解读时只引用脚本输出 + R3 已确认的命局画像，不掺入"你跟父母是因为命里 …"这种后视镜叙事

### 11.7 触发条件（避免 R3 滥用）

R3 是**条件触发**的：

- ✓ 用户在初次提问中提到「家庭 / 父母 / 父亲 / 母亲 / 原生家庭 / 出身 / 家世 / 我爸 / 我妈」等关键词 → 必须抛 R3
- ✓ 用户在分析过程中追问「我家怎么样 / 我爸是什么样 / 我妈呢」 → 必须抛 R3（在写 family 段之前）
- ✗ 用户没主动问家庭 → 不要抛 R3（也不要写 family 段）

不要把 R3 跟 R0/R1 混在一组里抛——R3 是独立的"原生家庭专项校验"。
