# 民俗志协议（Folkways Protocol）

> 民俗志（folkways）= 一个时代里"人们具体在用什么、玩什么、学什么、说什么"的物质和文化纹理。本协议规定 LLM 在 Step 3a 写区间叙事时如何引入这些细节，**用结构化约束防止"自由发挥"沦为编造**。

## 1. 设计哲学

LLM 训练数据里**已经有大量民俗志知识**（VCD 何时流行、Friends 哪年开播、苏联何时解体、AI 何时爆发），不需要硬编码数据库。但 LLM 缺的是：

1. **把民俗志和命理桥接起来的方法**（五行映射、阶级 prior 筛选）—— 这是项目原创，必须文档化
2. **对自由发挥的约束机制**（confidence、自检、援引规则）—— 防止 LLM 编造或臆想

本协议解决这两件事。

## 2. 民俗志的 6 层 sub-layer

每个 era_window 的民俗志应覆盖以下 6 层（写区间叙事时不必每层都写满，但 LLM 内部应过一遍）：

| Sub-layer | 关注什么 | 典型内容（中国 1990s 末为例） |
|---|---|---|
| **material** 物质生活 | 家电、通讯、交通、住房 | BP 机、VCD、21 寸彩电、自行车 → 摩托车 |
| **media** 媒介信息 | 广播、电视、报刊、互联网 | 央视新闻联播、读者、南方周末、BBS、QQ |
| **pop_culture** 流行文化 | 音乐、影视、偶像、文学 | 还珠格格、F4、王菲、痞子蔡、韩寒 |
| **fads** 热潮 | 学习、投资、创业、教育潮流 | 学英语、学计算机、股票热、考研、出国潮 |
| **communication** 沟通方式 | 主流通信、典型场景 | 写信、座机、IC 卡电话、QQ、第一波短信 |
| **rituals** 集体仪式 | 年度节庆、一次性大事件、日常仪式 | 春晚、香港回归、98 抗洪、新闻联播 |

**规则**：
- 写一个 era_window 的标志性细节时，应**跨层覆盖**至少 3 层（不要 5 条都谈物质）
- 每条标注其归属的 sub-layer（让 LLM 自检是否覆盖均匀）

## 3. 三档 confidence 机制（约束 ①）

LLM 在生成每条民俗志推论时**必须**标 confidence，否则该条不能写入输出。

### 3.1 三档判断标准

```
high 标准（满足 ≥ 2 条）：
  ✓ 全国/全地区性媒体集中报道
  ✓ 跨阶级跨城乡都有共同记忆
  ✓ 在主要历史叙事中被反复提及
  ✓ 即使不同政治立场也都承认这个现象存在

  示例（中国）：
    1997 香港回归 / 1998 还珠格格 / 2008 北京奥运 /
    2003 SARS / 2010 微博兴起 / 2020 疫情
  示例（全球）：
    1985 Live Aid / 2001 911 / 2008 金融危机 /
    2007 iPhone / 2022 ChatGPT

mid 标准（满足 ≥ 1 条）：
  ✓ 限定某个阶级 / 城乡 / 行业群体
  ✓ 流行有明显时间和地理梯度（一线先 → 三线后）
  ✓ 不同代际/地区的回忆颗粒度不同
  ✓ 是子文化但有清晰边界

  示例：
    BP 机（urban_middle 1996 / county 1999） /
    VCD（一线 1995 / 县城 1999） / 拨号上网 /
    2010 公众号红利 / 2003 博客 / 2014 微信红包

low 标准（任一条即为 low，禁止出现在输出）：
  ✗ 你（LLM）只能猜测但无法给出具体时间窗
  ✗ 该现象的存在本身就有争议
  ✗ 你不熟悉该地区/年代的具体情况
  ✗ 信息只来自单一来源 / 难以交叉验证

  → 转为开放问题："你那时候是不是在 X 城市 / 做 Y 工作？告诉我后我能更具体"
```

### 3.2 输出中的使用

| Confidence | 在输出中的位置 |
|---|---|
| **high** | 直接写入"标志性细节"或"首选形态" |
| **mid** | 写入"备选形态"，或在前面加条件（"如果你属于 X 群体..."） |
| **low** | **禁止**出现在断言中；可转为开放问题让用户提供更多信息 |

## 4. 五项自检表（约束 ②）

LLM 输出每条民俗志推论前，必须能回答 5 个问题。**任意 1 项答不出 → 不能进入"首选形态"**；**任意 2 项答不出 → 完全不能写**。

```
□ 1. 这个物件 / 事件的时间窗（first_appear / peak / declining）是哪一段？
     → 答不出 → confidence 自动降为 low → 不能写入输出
     
□ 2. 它的地域分布是什么？
     → 中国：tier1+tier2 / tier3+county / rural （3 档）
     → 全球：哪些国家 / 哪些区域
     → 答不出 → 必须在输出中注明"地域不确定"
     
□ 3. 它的阶级 marker 是什么？
     → 普及型 / 中产标志 / 精英标志 / 特定职业群体
     → 答不出 → 不能用作"首选形态"，只能用作"备选形态"
     
□ 4. 它的五行 tag 是什么？（按本协议 §5 推）
     → 答不出 → 不能参与"用神匹配筛选"，只能作为时代背景描述
     
□ 5. 它对应该用户的哪个命理事件？
     → 必须能说出"为什么这条民俗志值得提到"——
       是因为命理事件需要这种载体？还是仅仅作为时代背景？
     → 答不出 → 这条不应该写在 key_years 里，转为整段时代背景描述
```

**自检的强制性**：在 LLM 的内部 reasoning 中显式回答这 5 个问题（虽然不一定输出给用户），这是防止编造的根本手段。

## 5. 五行映射规则（项目原创桥接逻辑）

LLM 给民俗志条目打五行 tag 时，必须按以下规则推（**不是凭"印=印章=文书"那种传统取象自由发挥**）。

### 5.1 五行核心语义

| 五行 | 核心语义 | 现代物件示例 |
|---|---|---|
| **金** | 定型 / 收割 / 切割 / 结构化 / 高精度 | BP 机、Walkman、机械表、AI 芯片、单反相机、剃须刀、武器、外科手术 |
| **木** | 生发 / 教学 / 培育 / 知识 / 文化 | 学校、培训班、Coursera、知乎、读书会、医药、植物、文化产品 |
| **水** | 流动 / 连接 / 智识 / 信息 / 跨界 | BBS、QQ、邮件、微信、报纸、AI Agent、跨境业务、咨询、研究 |
| **火** | 被看见 / 演示 / 传播 / 表达 / 镜头 | 电视、电影、KTV、抖音、直播、Podcast、社交媒体、广告、灯光 |
| **土** | 稳定 / 承载 / 持有 / 收藏 / 不动产 | 房子、家具、长期持有股票、收藏、单位福利、土地、农业 |

### 5.2 复合 tag 规则

很多现代物件是多五行复合的——按"主功能 + 次功能"打 2-3 个 tag：

```
智能手机 = 金（硬件） + 火（屏幕传播） + 水（信息流）
VCD = 火（影像） + 金（光盘介质）
学英语（疯狂英语） = 木（教育） + 火（表达）
学计算机 = 金（机器） + 水（信息）
微博 = 水（信息流） + 火（公开传播）
公众号 = 木（文章/培育） + 水（传播） + 火（个人 IP）
直播带货 = 火（镜头） + 水（流量） + 金（成交）
比特币/区块链 = 金（密码学） + 水（流动性）
ChatGPT/AI Agent = 水（信息流） + 木（知识培育） + 金（结构化产出）
房产 = 土（不动产）
教培 = 木（教育） + 土（机构稳定）
出海跨境 = 水（流动） + 金（结算）
新能源车 = 火（电力） + 金（机械）
```

### 5.3 十神 tag 规则（辅助）

除五行外，还可标十神 tag，用于和命主当年的命理事件做更精确匹配：

```
食神/伤官类（产出 / 表达 / 才华）：
  - 学英语、学计算机、写公众号、搞自媒体、做产品

正官/七杀类（权威 / 体制 / 规则）：
  - 考公、进体制、单位变动、政策响应

正印/偏印类（学习 / 庇护 / 学问）：
  - 上大学、考研、读书会、师徒关系

正财/偏财类（资源 / 收益 / 实物）：
  - 投资、做生意、买房、副业

比肩/劫财类（同辈 / 协作 / 竞争）：
  - 创业合伙、同学会、同辈竞争
```

## 6. 阶级 prior 筛选规则

LLM 收到 `class_prior` 后，按以下映射筛选 folkways 候选：

```
urban_state_or_educated:
  prefer_class_marker: [urban_middle, urban_educated]
  prefer_region: [tier1, tier2]
  prefer_fads: [learning, education, cultural_consumption]
  典型场景：父母在体制 / 单位福利 / 子弟校 / 教育资源好

urban_market_oriented:
  prefer_class_marker: [urban_mass, urban_aspirational]
  prefer_region: [tier1, tier2, tier3]
  prefer_fads: [business, investment, side_hustle]
  典型场景：父母经商 / 跑业务 / 下海 / 早期接触市场

institutional_family:
  prefer_class_marker: [urban_middle, civil_service]
  prefer_region: [tier1, tier2, tier3]
  prefer_fads: [education, official_career]
  典型场景：父母公务员 / 国企管理 / 体制内晋升路径

rural_or_county:
  prefer_class_marker: [rural_aspirational, rural, county_mass]
  prefer_region: [tier3, county, rural]
  prefer_material: [摩托车, 黑白电视末班车, 缝纫机, 化肥/农资]
  典型场景：农村家庭 / 父辈务农或外出务工 / 县城

freelance_artisan_family:
  prefer_class_marker: [urban_mass, artisan]
  prefer_region: [tier2, tier3, county]
  prefer_fads: [craft, side_business]
  典型场景：手艺人家庭 / 个体工商户 / 自由职业

grassroots_self_made:
  prefer_class_marker: [urban_mass, mobile]
  prefer_region: any
  prefer_fads: [self_education, side_hustle, mobility]
  典型场景：草根 / 路径需自创 / 高度移动

early_high_pressure:
  prefer_class_marker: [single_parent, distant_parent, orphan]
  典型场景：单亲 / 失怙 / 留守 / 童年高压
```

**重要约束**：
- 阶级 prior 只用于**筛选 folkways 候选**，**严禁**作为输出 label 出现
- 一个命主的 class_prior 通常是多个 tag 的加权（如 {urban_state_or_educated: 0.5, urban_market_oriented: 0.3}），LLM 应混合考虑

## 7. 援引规则（约束 ③）

### 7.1 强制援引格式

每条民俗志推论在输出中必须包含以下信息（不一定显式列出，但读者能从文字中看出来）：

```
✓ 物件/事件名称（具体）
✓ 时间窗（年份范围 / 高峰年）
✓ 地域 / 阶级条件（如适用）
✓ 与命理事件的关联（"你这年走 X 命理事件，最可能体现为 Y 民俗形态"）
```

示例（合规）：

> 1998 ± 1 年是 VCD 入户、学计算机、笔友/萌芽杂志的高峰期（confidence: high）；你命局这年食神泄秀（+18），最可能的具体形态是：你在某个新工具/新媒介上找到了"被看见"的体验——打字比赛、机房作业被表扬、用 VCD 录下自己唱歌、给笔友写长信被回信。

### 7.2 禁止事项

- ❌ 凭空编造"那年流行 X"——如果 LLM 自己都说不出该现象的 5 个属性（§4 自检），不能写
- ❌ 把命理术语和民俗志细节混为一谈（"伤官见官 = 你买了 BP 机"——这是过度联想）
- ❌ 用未来视角倒推过去（"那年的 BP 机其实是 iPhone 的前身"——LLM 自我陶醉式叙事）
- ❌ 用刻板印象代替具体推论（"那年代农村就是穷"——必须给具体场景）

### 7.3 不熟悉时的诚实降级

如果某年/某区域 LLM 确实不掌握细节，**必须如实说**：

```
✓ "这一年的具体细节我不掌握，只能讲命理结构"
✓ "如果你能告诉我当时的城市/家庭背景，我能给更具体的推论"
✗ 瞎编"那年流行 X"
✗ 用模糊话术掩盖（"那时候大家都在 X"——必须给方向 + 维度）
```

## 8. handshake 民俗志锚点的特殊规则

民俗志锚点用于 Step 2.5 校验时，**比 key_years 更严格**：

- 必须 confidence=high 或 mid（不允许 low）
- 必须命主 yongshen 与该 folkways 条目 wuxing 至少有 1 个匹配
- 必须明确写"如果 X 没发生，这条就错"的可证伪点
- 限制 1 条/年份（不要密集轰炸用户记忆）

## 9. 用户反馈回流

用户在 handshake 中确认/否定的民俗志锚点，自动写入 `confirmed_facts.json`：

```json
{
  "folkways_corrections": [
    {
      "user_birth_year_range": [1980, 1990],
      "user_region_hint": "county",
      "era_window_id": "cn_reform_late_90s",
      "correction": "VCD 在县城真实普及高峰是 2000-2003，不是 1998",
      "confidence_after_correction": "mid",
      "reported_by": "user_xxx",
      "timestamp": "2026-04-21"
    }
  ]
}
```

LLM 下次推理时**必须读取**这个文件，作为校正性 prior。

## 10. 自检清单

写完一段含民俗志的解读后，按这个清单过一遍：

- [ ] 每条民俗志条目都标了 confidence
- [ ] 没有 low confidence 的条目出现在断言中
- [ ] 每条都过了 §4 五项自检
- [ ] 五行 tag 按 §5 规则打的，不是自由取象
- [ ] 援引格式按 §7.1（含物件 + 时间窗 + 地域/阶级 + 命理关联）
- [ ] 没有出现身份命名（"你出身工人阶级"等）—— 见 class_inference_ethics.md
- [ ] 不熟悉的部分诚实承认（不瞎编）
- [ ] 跨 sub-layer 覆盖 ≥ 3 层（不要 5 条都谈物质）
- [ ] 援引了 confirmed_facts.json 里的相关校正（如有）
